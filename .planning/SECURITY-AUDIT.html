Executive Summary

The most urgent risks are overpermissive Supabase RLS policies that allow any authenticated user to read/write data across ALL customers, combined with the anon key exposed in the frontend bundle. Together, these create a path to complete data theft. The Express API has strong auth middleware but several input validation gaps. 6 critical fixes have been applied.

Critical Findings

C-1: Overpermissive RLS on 7 Integration Tables

FIXED

Migrations 006-011

Any authenticated user (including customer portal users) could read/modify/delete ALL rows in pax8_mappings, inky_reports, threecx_mappings, threecx_reports, vultr_mappings, dmarc_report_mappings, vpentest_mappings. Policies used USING (true).

Fix: New migration restricts all 7 tables to admin + service_role only.

C-2: Settings Table Exposes API Secrets

FIXED

migrations/001_initial_schema.sql:744

Any logged-in user could read HaloPSA, Datto, JumpCloud, DarkWebID credentials stored in plaintext via the settings table.

Fix: Settings SELECT restricted to admin + service_role only.

C-3: cron_job_runs Readable by All

FIXED

migrations/004_add_cron_job_runs.sql:20

Error messages in cron logs could contain API credentials or internal system details visible to any authenticated user.

Fix: Restricted to admin + service_role.

C-4: Direct supabase.from() Calls Bypass Server

MITIGATED

LootITCustomerDetail.jsx, Pax8Config.jsx, hooks

Frontend makes direct DB inserts/deletes without server-side authorization. RLS is the only guard. Mitigated by RLS fix (C-1) but direct calls should be routed through server middleware long-term.

C-5: Supabase Anon Key in Frontend Bundle

EXPECTED

src/api/client.js, src/lib/app-params.js

Anon key is visible in JS bundle. This is expected for Supabase but dangerous with permissive RLS. Risk mitigated by C-1 RLS fix.

C-6: Hardcoded Production URL

FIXED

server/src/routes/users.js:90

Railway production infrastructure URL was hardcoded as fallback. Now throws error if FRONTEND_URL env var is not set.

C-7: Path Traversal in File Upload

FIXED

server/src/routes/upload.js:16

User-controlled filename used without sanitization. Now strips path separators, .., special chars, and validates MIME type against allowlist.

C-8: Content-Disposition Header Injection

FIXED

server/src/routes/functions.js:90

Unsanitized filename in response header. Now uses RFC 5987 encoding with sanitized filename.
