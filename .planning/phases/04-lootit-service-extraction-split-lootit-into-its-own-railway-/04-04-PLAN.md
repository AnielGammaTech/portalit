<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → document.cookie | Session persistence layer. Attacker with XSS can read/write cookies just like localStorage. Same threat model as today. |
| Browser → Supabase auth | Supabase validates the JWT signature server-side; cookie is only client-side persistence. |
| portalit.gtools.io ↔ lootit.gtools.io | Cross-subdomain cookie share via Domain=.gtools.io. Both origins are under our control. |
| Third-party site → .gtools.io cookie | Mitigated by SameSite=Lax + Secure. Browser refuses to set the cookie from off-site contexts. |
| CORS origin → backend | Backend allowlist validates Origin header via CORS_ORIGIN env var (verified: server/src/index.js:44-64). Adding a new origin is an env-var edit, not a code edit. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | src/lib/auth-storage.js — XSS exfiltration of cookie session | accept | Unchanged from current state. Same JS-accessible session as today's localStorage. Defense is React's default escaping + no dangerouslySetInnerHTML with untrusted data. CSP headers are out of scope for this phase. |
| T-04-02 | Spoofing / Session Fixation | document.cookie write with missing Domain scope | mitigate | Adapter hard-codes `Domain=.gtools.io` + `Secure` + `SameSite=Lax`. Grep-verified in acceptance criteria. Localhost branch is separate code path (no Domain attr — cannot leak to gtools.io). |
| T-04-03 | Tampering | CSRF via credentials:true + cookie auto-send | mitigate | Existing Express CORS middleware validates Origin against allowlist (verified server/src/index.js). The backend authenticates via `Authorization: Bearer` header, NOT from `req.cookies`. This must be re-verified during this plan — see Task 3's read_first list. |
| T-04-04 | Tampering / DoS | Oversize cookie value silently truncated by browser | mitigate | Chunking implemented in Plan 04-02 adapter (CHUNK_SIZE=3500, 10 chunks = 35KB ceiling). readFromCookie returns null on corrupted chunks → user re-logs, no silent compromise. |
| T-04-05 | Repudiation | Refresh token replay | mitigate | Supabase refresh tokens are single-use + rotated. Adapter always stores the newest (writeToCookie clears old chunks first). |
| T-04-06 | Information Disclosure | Stale localStorage session visible after cookie logout | mitigate | removeItem clears BOTH cookie and localStorage per D-22. Sign-out from either app propagates via onAuthStateChange SIGNED_OUT event. |
| T-04-07 | Spoofing | Domain confusion (gtools.io vs .gtools.io) | mitigate | Leading dot form is enforced as a const — no template string assembly with user input. |
| T-04-08 | Tampering / Session Loss | Multi-tab race during adapter swap (Pitfall 6 from RESEARCH) | accept | Documented risk per D-33. Backcompat dual-write minimizes drift window. Residual risk: a user with 2+ stale portalit tabs at the moment of deploy may need to re-login once. Acceptable UX cost. |
| T-04-09 | Denial of Service | Backwards-compat read returns stale session after kill-switch revert | accept | D-34 kill switch rolls back the createClient wiring; localStorage path remains intact. A brief window of "user sees stale session data" is acceptable — re-login resolves. |
| T-04-10 | Information Disclosure | CORS widening introduces new origin | mitigate | Backend CORS_ORIGIN env var is updated explicitly and ONLY by Plan 04-05 — not wildcard. Change is scoped to adding `https://lootit.gtools.io` (or the temporary Railway URL during verification). |
| T-04-11 | Repudiation / Safari ITP 7-day cap | Accept | accept | Documented in RESEARCH Pitfall 3. Safari ITP caps script-set first-party cookies at 7 days regardless of Max-Age. Users who are idle 7+ days must re-login. Mitigated naturally by Supabase refresh-token rotation for active users. |
</threat_model>
